# KAS: emit INFO log of configured mechanisms after startup

## Summary
ProblemThe KAS currently logs its full keyring configuration only at DEBUG level via a kas config entry:
{"level":"DEBUG","msg":"kas config","config":{"keyring":[{"alg":"hpqt:xwing","kid":"x1"},...]}}This makes it impossible to determine which cryptographic mechanisms a running KAS instance supports without either:
Enabling DEBUG-level logging (noisy in production)
Making HTTP probe requests to the KAS (/kas.AccessService/PublicKey) for each algorithm individually
Proposed SolutionAfter the KAS config is fully parsed and initialized, emit a single INFO-level log entry listing all configured key algorithms:
{"level":"INFO","msg":"kas initialized","mechanisms":["rsa:2048","ec:secp256r1","hpqt:xwing","hpqt:secp256r1-mlkem768","hpqt:secp384r1-mlkem1024"]}The mechanisms array should contain the algorithms currently supported by the loaded trust KeyManager or KeyServices. If preview features such as ec_wrapped_keys or hybrid_pqt_keys are disabled, the associated algorithms  (ec:secp… or mlkem…) must not be included in the list.
WhyTest infrastructure: xtest needs to detect PQ algorithm support before tests run to know which tests to skip. Currently this requires HTTP probing which is unreliable (wrong KAS URL, timing issues, key not yet created). A log entry is authoritative, zero-latency, and already in a known location.
Operators: At INFO log level, operators can immediately see what mechanisms their KAS instance supports — useful for incident response and configuration audits.
Standard practice: Health/readiness endpoints log their status at INFO. Mechanism capability is analogous configuration state.
Acceptance CriteriaKAS emits {"level":"INFO","msg":"kas initialized","mechanisms":[...]} after startup config parsing completes
The mechanisms array contains all algorithm strings from the configured keyring (deduped, no legacy duplicates)
Entry appears before the first request is served
xtest can be updated to read this entry from {PLATFORM_DIR}/logs/kas-km1.log to detect PQ support (tracked in companion PR on opentdf/tests)
