# Update Keycloak provisioning to use RFC 8693-compliant standard token exchange (V2) instead of preview feature

## Summary
BackgroundWith the upgrade to Keycloak 26+, the token-exchange feature is now available as a standard, RFC 8693-compliant feature (V2) rather than requiring the preview feature flag. The DSP Keycloak provisioning and compose configurations currently enable token exchange via the legacy preview mechanism. This ticket covers updating those references to use the standard V2 token exchange.
Related workDSPX-3397 — Add DPoP support to Java SDK (Keycloak v26 enforcement)
opentdf/platform#3582 — feat(core): Adds comprehensive DPoP (RFC 9449) support (branch DSPX-3397-platform-service)
Context from Slack threadThe platform PR (opentdf/platform#3582) bumps Keycloak to 26.2 in docker-compose. Testing against KC v26 confirmed all PEPs (Outlook, SharePoint, Virtru Desktop, Windows PEP) work except Gateway PEP, which has a known DPoP issue being addressed separately. Consensus is that KC 26 is acceptable to adopt once DPoP support lands in the Java SDK.
Current stateThe following files reference the legacy preview token-exchange feature:
dsp-compose.yaml
KC_FEATURES: "preview,token-exchange"internal/hsm/luna/scripts/test-luna-k8s-mode-all.sh
KC_FEATURES: "preview,token-exchange,admin-fine-grained-authz:v1"cmd/provision/provision_keycloak.go — uses fixtures.SetupCustomKeycloak() from opentdf/platform/lib/fixtures, which may also need updates to configure V2 token exchange realm settings.
Scope of workUpdate KC_FEATURES flags in compose files and test scripts to remove the preview and token-exchange flags (or replace with the V2-equivalent feature name if Keycloak requires an explicit flag).
Update Keycloak provisioning code (cmd/provision/provision_keycloak.go and/or fixtures.SetupCustomKeycloak) to configure the realm for standard RFC 8693 token exchange (V2) instead of the preview token-exchange feature.
Bump Keycloak version in DSP compose files to 26.2+ (coordinated with DPoP support landing).
Validate that all PEPs continue to function correctly with the V2 token exchange:
Outlook PEP
SharePoint PEP
Virtru Desktop PEP
Windows PEP
Gateway PEP (pending DPoP fix in Java SDK — DSPX-3397)
Update any documentation referencing the preview token-exchange feature flag.
Acceptance criteriaDSP no longer uses the preview Keycloak feature flag for token exchange
Token exchange is configured using the standard RFC 8693-compliant V2 mechanism
Keycloak version in compose files is bumped to 26.2+
E2E tests pass with the updated configuration
PEP compatibility validated (Gateway PEP may depend on DSPX-3397 completion)
