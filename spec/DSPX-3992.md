# CI: make start-up-with-containers key/config assembly principled (remove version/PQC heuristics)

## Summary
Backgroundopentdf/platform's test/start-up-with-containers/action.yaml (the reusable "spin up platform" CI action consumed by opentdf/tests xtest) assembles the KAS keyring + cryptoProvider.standard.keys into opentdf.yaml. A regression in opentdf/platform#3594 (DSPX-3499) put an early exit 0 in the same step that generates opentdf.yaml, so any platform ref lacking PQC keygen (e.g. released tags like service/v0.9.0, which predate service/cmd/keygen) never created opentdf.yaml and every later step failed with stat opentdf.yaml: no such file or directory. This broke all main/lts xtest runs.
Quick fix (already up): opentdf/platform#3750 removes the misplaced guard. Verified via opentdf/tests#559 (dispatched run with platform-ref=lts → v0.9.0): the Check out and start up platform step now passes. This ticket tracks the durable cleanup so the bug class can't recur.
Problem (why this is tech debt)Even after the quick fix, the action conflates three concerns and relies on two fragile heuristics:
opentdf.yaml is still created inside the extra-keys step — config creation is coupled to key logic; any future early-return in that step reintroduces the same bug.
Two hardcoded key inventories (base keyring in "Map the config"; PQC list in "Enable PQ") that drift from what init-temp-keys.sh/keygen actually produce — keygen emits 5 PQC key types, the action wires up 3.
Version awk gate (PLATFORM_VERSION >= 0.7.1 → allow rsa:4096 ec:secp384r1 ec:secp521r1) for EXTRA_KEYS — a hardcoded version→capability map that rots.
Silent degradation — init-temp-keys.sh is #!/bin/sh with no set -e, so a missing service/cmd/keygen is swallowed; the only signal was the downstream crash.
Proposed fixAlways materialize base config (decouple). Dedicated first step: cp opentdf-dev.yaml opentdf.yaml; every later step edits in place with yq -i. Makes "opentdf.yaml missing" structurally impossible and each augmentation independently guardable.
Build the keyring from file existence, declaratively. One table of candidate keys (kid, alg, private_file, public_file) covering base + PQC; include each iff its private file exists on disk. openssl always writes ec/rsa; keygen writes PQC iff the ref supports it — so this single rule replaces both the hardcoded base list and the PQC special-case step, and sets hybrid_tdf_enabled iff ≥1 PQC key is included. No version logic; file existence is a genuine capability probe (same binary generates and consumes).
Gate EXTRA_KEYS by a real capability probe, not version. Preferred: add a small platform command sourced from lib/ocrypto/key_type.go (e.g. extend service/cmd/keygen with list --json of supported wrapping algs). The action runs it against platform-ref and builds the allow-list from the output; old refs lacking the command fall back to the conservative base set (ec:secp256r1, rsa:2048) — correct, since those refs reject newer algs anyway. This removes the awk entirely. (Alternatives: per-key minPlatformVersion in xtest/extra-keys.json; or a --validate-config dry-run + drop-loop — heavier, needs new platform code.)
Harden init-temp-keys.sh to explicitly detect a missing service/cmd/keygen and log "PQC keygen unavailable for this platform-ref; skipping" instead of relying on a naked go run under no-set -e.
Also apply the step 1–2 pattern to the sibling test/start-additional-kas/action.yaml (same inventory pattern; #3594 patched it with PQC_ENABLED=false).
SequencingSteps 1, 2, 4 are CI-only (action YAML + shell), no Go changes, and remove the coupling bug class and the PQC special-casing — a self-contained PR that supersedes the #3750 quick fix.
Step 3 (capability probe) is a small separate opentdf/platform change (keygen list); until it lands, the version awk stays as a documented fallback.
Acceptance criteriaopentdf.yaml is always generated regardless of which key files exist.
Keyring/keys entries are derived from on-disk key files (no hardcoded per-alg lists that can drift).
No PLATFORM_VERSION version-number heuristic remains for algorithm gating (or it exists only as a documented fallback when the capability probe is unavailable).
xtest passes across platform-ref in {lts (v0.9.0, no PQC), latest, main} for go/java/js.
