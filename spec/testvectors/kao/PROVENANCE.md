# BaseTDF KAO Test Vectors — Provenance

This directory contains conformance test vectors for the BaseTDF Key Access
Object. Each vector is a JSON file under `vectors/`, indexed by `index.json`,
and validated against `vector.schema.json`.

## Status

The vector schema and inventory (`index.json`) are stable. The vector content
under `vectors/` is being authored in two waves:

1. **Wave 1 (Phase 1, in this PR):** five hand-verified high-value vectors
   covering one positive case per algorithm family plus the
   `neg-policy-binding-tampered-001` negative case. Spot-checked by hand
   decoding and cross-referenced against the prose spec in
   [`../../basetdf/basetdf-kao.md`](../../basetdf/basetdf-kao.md).

2. **Wave 2 (Phase 2):** the remaining vectors, generated by the reference
   Python validator at [`../../tools/python/`](../../tools/python/) using
   deterministic seeds. The CI job `fixtures-fresh` verifies that the
   regeneration script produces byte-identical output to the checked-in files.

Until both waves land, vector files referenced by `index.json` may not yet
exist on disk. The Python test harness skips missing vectors and reports them
as `pending`.

## Test keys

Test keys live under `keys/`. Each key file has a companion `<id>.meta.json`
carrying:

- `id` — the key identifier referenced from vectors via the `inputs.keys` map.
- `algorithm` — one of `RSA-2048`, `RSA-4096`, `EC-P256`, `EC-P384`,
  `ML-KEM-768`, `ML-KEM-1024`, or `hybrid`.
- `provenance` — the seed and generation procedure (see below).
- `WARNING` — the literal string `"NON-PRODUCTION TEST KEY — DO NOT USE"`.

### Generation procedure

Test keys are generated deterministically from fixed seeds via
[`../../tools/python/scripts/gen_test_keys.py`](../../tools/python/scripts/gen_test_keys.py)
(added in Phase 2). The seeds are documented per-algorithm in the script's
docstring. RSA keygen is deterministic via a per-call RNG shim that wraps
`os.urandom` with a `ChaCha20`-based deterministic stream keyed by the seed.

The CI job `fixtures-fresh` regenerates the keys into a temporary directory
and asserts byte-identical output via `diff -r`. If the script becomes
non-deterministic (e.g., due to a `cryptography` upgrade that changes RSA
keygen internals), the CI job fails and surfaces the drift; the fix is to
re-pin the cryptography version or update the checked-in keys.

## Security note

The keys and DEK shares in this directory are **public** (committed to a public
repository) and exist solely to make conformance vectors reproducible. They
MUST NOT be used in any production system, test harness, or training pipeline
that handles real data. Each key file carries an explicit `WARNING` field as
a defense-in-depth measure against accidental copy-paste into production code.

This data is not subject to private-key disclosure rules: the corresponding
"private" keys are deliberately public test fixtures, equivalent to the test
keys in NIST CAVP and IETF specifications.
